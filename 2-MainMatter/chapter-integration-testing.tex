\chapter{Integration and Testing}
\label{cha:integration-testing}

This chapter describes how the components developed during this internship, the
\gls{sdr-control-system} and the \gls{rs232-interface}, were brought together
into a working system, the environment in which that system was validated, and
the testing strategy used to ensure its correctness and reliability.
\section{Unit Testing}

Before integrating the components, unit tests were developed for both the
\gls{sdr-control-system} and the \gls{rs232-interface}. These tests were
designed to validate the functionality of individual components in isolation,
ensuring that each component behaved as expected under a variety of conditions.
For the \gls{sdr-control-system}, unit tests were created to verify the correct
generation of \gls{GNSS} signals, the proper handling of configuration
parameters, and the correct interaction with the underlying \gls{SDR} hardware.
For the \gls{rs232-interface}, unit testing was inherently more limited because
correctness depends not only on the application code, but also on serial
hardware behaviour, operating-system networking support, and peer
configuration. As a result, the most meaningful validation of the serial path
was performed through end-to-end tests using the external tester described in
\autoref{sec:the-external-tester}.

Code coverage for the unit test suites was measured using
\texttt{gcovr}, generating coverage reports from the instrumented test
binaries, which were then ingested by a SonarQube instance for
continuous tracking across commits. This provided visibility into
coverage trends as new functionality was added throughout the
internship, and helped identify modules and branches that lacked
adequate test coverage, particularly around error-handling paths that
were not always exercised by the otherwise functionality-focused unit
tests. It should be noted that the coverage figures obtained are
treated only as an indicator of test thoroughness rather than a target
pursued for its own sake: a high coverage percentage does not by
itself guarantee correctness, and some low-level hardware interaction
paths remained difficult to exercise through unit testing alone, which
is part of the motivation for the end-to-end testing described in
\autoref{sec:testing}.

\section{Environment Overview}
Throughout this chapter, references to ``the computer'', ``the host'', or ``the
generator'' refer specifically to the machine running the \gls{GNSS} signal
generator software stack, which hosts both the \gls{sdr-control-system} and the
\gls{rs232-interface}, and to which the \gls{SDR} hardware (DTA-2116 PCIe
cards) is physically attached.
It is important to frame the environment in which this system was developed and
validated as a prototype setup, distinct from a final deployment environment.
In a deployed scenario, the generator would be placed in a loopback, over a
\gls{RF} uplink to a remote receiver that would measure the transmitted signal,
and calculate any needed adjustments to the signal generator's configuration.
That remote receiver would then communicate those adjustments back to the
signal generator over the \gls{RS232} link. In the final architecture, that
serial link carries \gls{PPP}, allowing the control path to operate over IP and
gRPC even when the physical medium is serial. In this prototype environment,
however, such setup was too complex and impractical to maintain for day-to-day
development and testing, and so a testing component was developed, as described in
\autoref{sec:the-external-tester}. The external tester simulates the behaviour
of the remote receiver and any other measurement equipment that would normally
be present in a deployed scenario.
\autoref{fig:prototype-test-environment} illustrates this prototype
environment, showing the generator and the external tester connected via
the simulated \gls{RF} uplink and the \gls{RS232} feedback loop.

\section{The External Tester}
\label{sec:the-external-tester}

The external tester is a hardware/software component that simulates the
behaviour of the remote receiver and any other measurement equipment that would
normally be present in a deployed scenario. It is used to validate the
\gls{rs232-interface} and the \gls{sdr-control-system} in a controlled and
repeatable manner. The setup is quite simple, the generator is connected to
measurement equipment over a \gls{RF} link, which relays information to the
tester. The tester then communicates with the generator over a
\gls{PPP}-over-\gls{RS232} link, issuing gRPC requests and observing responses
that are representative of what the real equipment would trigger. This allows
both the \gls{rs232-interface} and the \gls{sdr-control-system} to be
exercised end-to-end, covering serial link configuration, \gls{PPP} transport,
network reachability, gRPC invocation, and signal generation. This allows the
behaviour of the real system to be validated under controlled and repeatable
conditions, without depending on the availability of physical instrumentation.


\section{Integration}
\label{sec:integration}

During integration with the existing proprietary systems at GMV, the original
idea of maintaining a dedicated application protocol on top of \gls{RS232} was
reconsidered. Once the serial path was understood primarily as a transport
constraint rather than an application constraint, it became more attractive to
reuse the already existing gRPC interface instead of maintaining two parallel
control planes.

The adopted solution was to establish a \gls{PPP} link over \gls{RS232},
thereby converting the serial connection into a standard point-to-point IP
channel. The gRPC client-server boundary described in
\autoref{sec:sdr-design-and-implementation} therefore remained relevant and was
exercised through that channel rather than being bypassed. This reduced custom
protocol code, aligned the serial integration path with the rest of the system,
and allowed existing networking tools and workflows to be reused.

This change also illustrates a useful design lesson. The important abstraction
was not that \gls{RS232} should have its own bespoke command language, but that
the control interface should be transport-agnostic. Once the serial link was
made to look like an IP network, the higher layers could remain unchanged.

\section{Testing}
\label{sec:testing}

While the components of the \gls{sdr-control-system} and the
\gls{rs232-interface} were validated individually through unit and
integration tests during development, as described throughout this
report, end-to-end verification against external reference
instrumentation was also performed to confirm that the system behaves
correctly when driving real \gls{SDR} hardware. This section describes
three test cases used for this purpose, each exercising the system
against a different piece of reference equipment connected to the
output of a DTA-2116 card.

\subsection{Test Case 1: Power Configuration}
\label{sec:tc1-power}

\textbf{Objective:} verify that configured transmit power matches the
power measured at the \gls{SDR} output, and that start/stop and
reconfiguration behave correctly.

\textbf{Setup:} DTA-2116 \gls{RF} output connected to a Rohde \&
Schwarz NRQ6 power sensor.

\begin{center}
\small
\begin{tabular}{|c|p{5.5cm}|p{4.0cm}|}
\hline
\textbf{Step} & \textbf{Action} & \textbf{Expected Result} \\
\hline
1 & Configure power $P_1$ and \texttt{start}. & Measured power
matches $P_1$ within tolerance. \\
\hline
2 & \texttt{stop}. & Measured power drops to noise floor. \\
\hline
3 & Reconfigure to $P_2$ and \texttt{start} again. & Measured power
updates to $P_2$, confirming the new configuration is applied. \\
\hline
\end{tabular}
\end{center}
